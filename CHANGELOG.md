# Changelog

All notable changes to **i-muslim** will be documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/), and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

## [Unreleased]

### Changed
- **Admin hadith JSON export is now translator-friendly: `_instructions` block at the top + `[TRANSLATE TO …]` placeholders for every missing translation slot.** Non-technical translators receiving an admin export couldn't tell where to type — empty translations were just absent keys. Updated [lib/hadith/export.ts](lib/hadith/export.ts) so every hadith's `translations` object always carries an entry for each requested language: filled when content exists, otherwise a clear `"[TRANSLATE TO Russian (Русский): replace this string with the translation]"` placeholder. With `?lang=all` (the default) every non-Arabic language in `ALL_LANGS` is materialised; with `?lang=ru` (or any specific code) only that language appears. A new top-level `_instructions` block lists the purpose, the rules (do-not-modify list), the language-code legend, and a per-field key so translators know which fields are read-only reference vs. editable. Verified with a one-off `tsx` script: `lang=all` emits four placeholder slots for the missing en/ru/az/tr; `lang=ru` emits a single Russian slot, leaving the file focused. No route or dialog changes — same `/api/admin/hadith/[collection]/download` and `/api/admin/hadith/[collection]/book/[book]/download`, same admin popover. Type-check clean.

### Removed
- **Public-side hadith JSON download buttons + their API routes.** The collection-level and book-level "Download (JSON)" buttons on `/hadith/[collection]` and `/hadith/[collection]/[book]` are gone, along with the [components/site/hadith/DownloadJsonButton.tsx](components/site/hadith/DownloadJsonButton.tsx) client component, the public route handlers `/api/hadith/[collection]/download` and `/api/hadith/[collection]/[book]/download`, the cached `getHadithsByCollection` fetcher in [lib/hadith/db.ts](lib/hadith/db.ts), and the `hadithDownload.*` i18n namespace (sync:locales propagated the removal to ar/id/tr). The shared serializer at [lib/hadith/export.ts](lib/hadith/export.ts) loses its `mode` option — admin downloads always emit drafts + `published` / `publishedTranslations` / `notes`, which was the only behaviour still being exercised. Admin downloads (`/api/admin/hadith/[collection]/download` and `/api/admin/hadith/[collection]/book/[book]/download`, with `?lang=` and `?chunked=1`) are unchanged. Build, lint, and locale parity all clean after the deletion.

- **AI Translator on hadith / Quran: bumped `maxOutputTokens` 2048 → 8192 + actionable diagnostics replace the generic "Gemini returned an empty response." toast.** A translator clicked **Translate with AI** on a hadith and got the bare empty-response message with no clue why. The string in [lib/admin/ai/gemini-translate.ts](lib/admin/ai/gemini-translate.ts) was the catch-all for *every* path where `result.response.text()` was empty — the SDK collapses prompt blocks, RECITATION blocks, SAFETY blocks, MAX_TOKENS truncation, and missing candidates all to `""`. After landing the diagnostics, the same hadith re-surfaced as `finishReason: "MAX_TOKENS"` — the actual root cause was budget truncation, not the RECITATION block I'd hypothesised. Gemini 2.5 models charge thinking tokens against `maxOutputTokens`, so 2048 was tight: the model burned most of its budget reasoning before any visible translation emerged. Bumped to 8192 — comfortable headroom for the realistic worst-case sacred-text translation while staying well under model caps (Flash supports 65536). The legacy `@google/generative-ai` v0.24 SDK doesn't expose `thinkingConfig`, so disabling thinking entirely would require migrating to `@google/genai` — out of scope. Diagnostics: `response.promptFeedback.blockReason` and `response.candidates[0].{finishReason,safetyRatings}` are inspected before returning, with a specific message per case (`RECITATION` → suggest a different model in /admin/settings or manual translation; `SAFETY` → name the flagged categories+probabilities; `MAX_TOKENS` → suggest retrying or shorter source; prompt-blocked → name the blockReason). Also passes `safetySettings` with all four standard categories at `BLOCK_ONLY_HIGH` — sacred-text input is canonical, not user-generated, and legitimately mentions warfare / hudud / marriage; the previous default-MEDIUM thresholds were the wrong calibration for this codepath (RECITATION is a separate filter unaffected by safetySettings). A `console.error("[gemini-translate] empty response", { …, usageMetadata })` line preserves full diagnostics server-side (model, finishReason, promptFeedback, safetyRatings, token counts — never the API key or prompt body) since the translator only sees the toast. Same `translateSacredText` is used by the Quran translate action ([app/\[locale\]/(admin)/admin/quran/_actions.ts](app/[locale]/(admin)/admin/quran/_actions.ts)) so both flows benefit. `npm run lint` and `npx tsc --noEmit` clean.

After the patch, `curl -sI http://localhost:7777/profile/matrimonial` returns `Location: /en/profile/matrimonial` (relative — Next 16 normalises since the redirect now matches the request host:port), so the browser preserves the dev port. - **Visiting `https://i-muslim.com` in production redirected to `https://i-muslim.com:8080/en` and timed out (`ERR_CONNECTION_TIMED_OUT`).** Firebase App Hosting runs the Next.js container on Cloud Run, which terminates external 443 traffic and forwards it to the container on internal port 8080 — but `request.nextUrl` inside the proxy middleware reflects that internal request, so any absolute redirect built from it leaked `:8080` into the `Location` header. Both the two custom redirects in [proxy.ts](proxy.ts) (unauthenticated `/admin/*` → `/<locale>/login`, logged-in user on `/login` → `/<locale>/admin`) and next-intl's `/` → `/<locale>` redirect were affected. - **Shared `<CountryCombobox>`** (mus-1193) — searchable, alphabetically sorted dropdown of all ~250 countries, localized into en/ar/tr/id via `i18n-iso-countries`. Replaces hand-rolled country `<Input>` fields across the profile, matrimonial preferences, mosque admin form, public mosque submission, business editor, certification-bodies admin, and the matrimonial admin filter. Stores ISO-3166 alpha-2 codes; supports single-select and multi-select (chips) modes.
- `npm run migrate:country-iso2` — one-shot Firestore migration that converts legacy free-text `profile.country` and `matrimonialProfiles.*.country` values to ISO-2 codes. Idempotent; supports `--dry-run`.

### Changed
- Profile schema tightened: `country` is now `^[A-Z]{2}$` (or empty) instead of free-text 2–60 chars. Matrimonial `preferredCountries` is now a `string[]` of ISO-2 codes instead of a comma-separated text field.
- **Profile + Matrimonial merged.** Identity and deen fields (display name, gender, DOB, country/city, languages, madhhab, sect, prayer commitment, hijab/beard, education, profession, marital history, bio) now live on `/profile` as a single editable form and persist to `users/{uid}.profile`. Matrimonial-only fields (preferences, polygamy stance, photos) are gated behind an opt-in toggle on the new `/profile/matrimonial` page, which also hosts the inbox/matches view. `/matrimonial/onboarding`, `/matrimonial/settings`, and `/matrimonial/inbox` now redirect to the unified surface.
- Profile sidebar restyled as a rounded card inside the centered content area (sticky just below the top nav) instead of a full-bleed left rail. Sidebar nav trimmed to Overview / Reading / Favorites / Matrimonial — the four "Your space" link cards are gone, replaced by the inline editable profile form.

### Added
- Heart/Bookmark "save" button on individual Ayah, Surah header, Hadith, Article (card + detail), Event detail, and Matrimonial profile detail (bookmark variant — distinct from the existing Express-Interest heart). Anonymous clicks show a sign-in toast; signed-in clicks toggle a Firestore favorite optimistically.
- Reading progress auto-capture: an `IntersectionObserver` on Surah and Hadith book pages records the most-visible item (with a 2-second visibility threshold) and flushes it via `navigator.sendBeacon` to `/api/profile/reading-progress` on `visibilitychange === 'hidden'` or `pagehide`. Stored at `users/{uid}/state/readingProgress`.

### Fixed

### Changed

The admin form now has a single Title field and a single Description field; the public site renders that single value regardless of the surrounding UI locale.

### Added
- Activated reserved locales now show a **completion percentage chip** on each row in /admin/settings → Interface. Click the chip (or "Edit") to open a phrase-by-phrase editor that lists every key with the base value above and an editable input below; supports search, "untranslated only" filter, and bulk-save (only changed rows are written). Updates land at `config/uiLocales/locales/{code}.messages` via a new `updateUiLocaleMessagesAction` server action.

### Changed
- Activate-locale dialog widened to a two-column layout (metadata fields stacked on the left, translations JSON textarea filling the right column). The Direction field is gone — RTL is auto-derived from `RTL_LOCALES`, since whether a script is right-to-left is a property of the language code, not something the admin should toggle.
- Settings: each row's toggle is now the activation affordance for unactivated reserved locales (the separate "+ Activate" button is gone). Toggle reads OFF until activated; clicking opens the paste-JSON dialog. Once activated the toggle controls public-switcher visibility as before, and the locale is auto-included in `uiEnabled` so the toggle reads ON without a second click. The descriptive paragraphs above each tab's list have been removed.
- Admin Settings page is now tabbed (Interface / Qur'an / Hadith); the Interface tab merges the bundled and reserved locale lists into one. Reserved-locale rows show an "Activate" button when inactive and "Edit" + "Deactivate" once activated.
- "Download base JSON" button in the activate-locale dialog so admins can grab the latest `messages/<base>.json` to translate offline before pasting back.
- `npm run sync:locales` script: mirrors activated reserved-locale Firestore docs to the current bundled-locale shape, adding missing keys with English placeholders and dropping stale ones. Idempotent. CLAUDE.md instructs Claude to run it after any change to `messages/*.json` that adds, renames, or removes keys.
- Reserved-locale pool (`ru`, `az`, `fr`, `ur`, `fa`, `bn`, `ms`, `de`, `es`, `hi`) routable from day one. Admin can "Activate" any of them from /admin/settings → Reserved interface locales by pasting a translations JSON; the locale's messages live in Firestore (`config/uiLocales/locales/{code}`) and are deep-merged over the base locale at request time, so partial uploads fall back gracefully.
- Turkish (`tr`) Qur'an/Hadith translation language with Diyanet İşleri Quran translation. Run `npm run seed:quran:lang -- --lang=tr` and `npm run seed:hadith:lang -- --lang=tr` to populate text, then enable in /admin/settings.
- Per-language seed scripts (`seed:quran:lang`, `seed:hadith:lang`) — additive, idempotent, preserve admin-edited translations via `editedTranslations.<lang>` flag.
- Admin Settings page: enable/disable UI locales and Qur'an/Hadith translation languages; disabled entries are hidden from public switchers without affecting routing.

### Changed
