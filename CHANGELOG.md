### Fixed
- **Visiting `https://i-muslim.com` in production redirected to `https://i-muslim.com:8080/en` and timed out (`ERR_CONNECTION_TIMED_OUT`).** Firebase App Hosting runs the Next.js container on Cloud Run, which terminates external 443 traffic and forwards it to the container on internal port 8080 — but `request.nextUrl` inside the proxy middleware reflects that internal request, so any absolute redirect built from it leaked `:8080` into the `Location` header. Both the two custom redirects in [proxy.ts](proxy.ts) (unauthenticated `/admin/*` → `/<locale>/login`, logged-in user on `/login` → `/<locale>/admin`) and next-intl's `/` → `/<locale>` redirect were affected. Added a `fixForwardedLocation` post-processor that rewrites the `Location` header using the `x-forwarded-host` / `x-forwarded-proto` headers Cloud Run sets, and renamed the existing `proxy()` body to `proxyImpl()` so a thin `proxy()` wrapper can apply the fix to every return path in one place. Local dev is unaffected (no `x-forwarded-host` → helper short-circuits and leaves `localhost:3000` redirects untouched). No config-side fix was available — Next.js 16 has no `trustHostHeader`-equivalent setting.
- **Admin bell-dropdown notifications that link to the deleted `/admin/mosques/queue` route (or to the transitional `/admin/mosques?submission=<id>` shape) no longer 404 — they're rewritten to `/admin/mosques` at the read boundary.** Old notification docs in Firestore still carry `link: "/admin/mosques/queue"` from before commit `108ae85` deleted that page (the Moderation Queue that got folded into the main mosques list under the Pending Review filter), and a smaller cohort still carries `?submission=<id>` from the in-between window before submissions were unified into the `mosques` collection (commit `1069697`). The notification creation code is already correct (`/admin/mosques?slug=<slug>`), so this only affects historical rows. Added a small `remapStaleAdminLink()` helper in [lib/admin/data/notifications.ts](lib/admin/data/notifications.ts) called from `normalizeNotification` — it rewrites those two known-dead patterns to `/admin/mosques` and passes everything else through. No DB writes, no new routes, no client-side changes.

- **`/admin/mosques` was silently missing rows that the public `/mosques` page showed.** The admin list query at [lib/admin/data/mosques.ts:254](lib/admin/data/mosques.ts#L254) uses `orderBy("updatedAt", "desc")`, and Firestore drops any document missing the ordered-on field from the result set. Mosques imported through paths that didn't set `updatedAt` (older seeds, manual Firestore console edits, CSV imports prior to the field being introduced) appeared on the public site (which queries `where("status", "==", "published")` with no `orderBy`) but never reached the admin table. Added a one-shot backfill script [scripts/migrate-mosque-updated-at.ts](scripts/migrate-mosque-updated-at.ts) (registered as `npm run migrate:mosque-updated-at`, `--dry-run` supported) that scans the `mosques` collection and writes `updatedAt = createdAt` for any doc with a `Timestamp` `createdAt`, falling back to `Timestamp.now()` otherwise. Idempotent — re-runs skip docs that already have `updatedAt` set. Mirrors the existing migration pattern in [scripts/migrate-hadith-publish-status.ts](scripts/migrate-hadith-publish-status.ts) (same env loader, batch size, dry-run flag).
- **Reopening the ayah/hadith comments popup after posting showed "Be the first to comment" even though the icon badge counted the user's comment.** [CommentsPopupButton](components/comments/CommentsPopupButton.tsx) loaded the thread on first open and cached the result in component state with a `if (!data) load()` gate, so subsequent opens re-rendered from the stale `data` (the empty fetch from before the post) — meanwhile the just-posted comment lived only in `CommentList`'s internal state, which was thrown away when the dialog closed. Two fixes: (1) `CommentsPopupButton` now refetches on every open, not just the first, so any state that changed elsewhere (other tabs, admin moderation) is reflected; (2) `CommentList` now accepts a `seedComments` prop + `onCommentCreated` callback. The popup wrapper holds the seed list across mount cycles, so the user's just-posted comment is rendered (deduped against the fetched page) whether or not the API returns it — important while the new composite indexes from the previous fix haven't been deployed yet.
- **Just-posted comment replies didn't appear in the thread until a refetch — and the refetch came back empty.** Two issues stacked: (1) `CommentItem` dispatched a `comments:reply-created` window event to inject the new reply optimistically, but `CommentReplyList` only mounts AFTER `setShowReplies(true)` re-renders the parent — its event listener attaches too late, so the freshly-posted reply was lost.
